# Right-BICEP

## 核心思想

来自单元测试圈的检查清单(Hunt & Thomas《Pragmatic Unit Testing》),
**不是经典 spec-based testing 方法论之一**——它是一个辅助 checklist,
专门补抓"测试看似通过但其实没真验证"这类问题。

六个字母:

- **R**ight — 结果对吗?
- **B**oundary — 边界条件考虑了吗?
- **I**nverse — 能否用反向操作验证?
- **C**ross-check — 有别的途径独立验证吗?
- **E**rror — 错误处理测了吗?
- **P**erformance — 性能可接受吗?

注意:**R/B/E 三个字母与经典方法论(EP/BVA/Use Case)重叠**,
所以 Inspector 在用 Right-BICEP 时**只关注 I / C / P 三个字母**。

## 选择性激发条件

Inspector **不**对所有功能都用 Right-BICEP。当且仅当规约满足以下条件时激发:

| 条件 | 激发字母 | 含义 |
|-----|--------|----|
| Behaviors 含写入语义(创建/修改/删除/状态转移) | **I** | 操作有副作用,需要反向验证 |
| Behaviors 中 server_state_after 字段非空 | **C** | 服务器有可独立验证的状态 |
| 规约 invariants 或 hints 提到时间限制 | **P** | 有性能预期可断言 |

**三个条件可叠加**——一个功能可能同时激发 I + C(如下单功能)。
**三个条件都不满足**(如纯展示功能、纯前端组件、纯静态页面),**完全跳过 Right-BICEP**。

### 为什么要选择性激发

这三个字母对应的检查在不适用场景下会变成噪音:

- 纯查询功能没有"反向"概念——Inverse 字母用不上
- 单一数据源的功能没有"另一条路径"——Cross-check 用不上
- 后台批处理用户感知不到——Performance 用不上

硬上会让 Inspector 强行挑刺,生成低质量建议。

## 各字母检查方式

### I — Inverse(反向验证)

- 操作完成后,能否用反向操作验证?
- 注册后 → 能用注册的账号登录吗?
- 删除后 → 列表里真的没了吗?
- 加车后 → 购物车里真的有这个商品吗?

**这是最容易漏的检查**。绝大多数测试只断言"操作返回成功",不验证"系统真的进入了预期状态"。

### C — Cross-check(交叉验证)

- 同一个事实,能否通过另一条独立路径验证?
- 用户在 UI 改了头像 → 通过 API 查这个用户,头像 URL 应该一致
- 用户加了商品 → 在另一个浏览器登录同账号,购物车应该同步

**Cross-check 抓 UI vs 数据不一致、缓存陈旧等 bug**。

### P — Performance(性能)

- 规约或业务对响应时间有期望吗?
- 测试有没有验证?
- 例:登录应该 < 2s 完成,但用例完全不看时间——P 字母漏了

## Inspector 反馈样例

```
P0-001: Inverse 检查全面缺失
- Methodology: Right-BICEP (I)
- Issue: 注册流程的所有 TC 都只断言"注册成功页面出现",没有任何 TC 后续尝试用注册的账号登录
- Suggested fix: 增加至少 1 个 TC,注册成功后立刻执行登录,验证账号真的可用

P1-001: Cross-check 缺失
- Methodology: Right-BICEP (C)
- Issue: 用户更新头像的 TC 只看了 UI 上头像变了,没通过 API 或刷新页面验证服务器侧真的更新
- Suggested fix: 补一个 TC,更新后通过 API 调用获取用户信息,确认头像 URL 已变更

P1-002: Performance 期望未验证
- Methodology: Right-BICEP (P)
- Issue: 规约提到"登录应在 2 秒内完成",但所有 TC 都没有时间断言
- Suggested fix: 在登录类 TC 中加入 duration < 2s 的断言
```

## 严重等级判定

- **P0**:激发的字母对应的检查**全面缺失**(如激发了 I 但整个测试集没有任何反向验证)
- **P1**:激发的字母对应的检查**部分覆盖**(部分 TC 该有但没有)
- **P2**:具体 TC 可以补一个 cross-check 增强

## 不要重复提

- R/B/E 字母的问题 → 应该在 EP/BVA/Use Case 阶段就提了。Right-BICEP 这里**只针对 I/C/P** 提反馈
- 如果激发条件全不满足,Right-BICEP 整个跳过,**不要在 Self-Check 表中列出**

## 适用场景速查

| 功能类型 | 是否激发 | 激发字母 |
|-------|--------|--------|
| 注册/登录/登出 | ✅ | I + C(写入 + 服务器状态变化) |
| 加车/下单/支付 | ✅ | I + C + 可能 P |
| 用户编辑资料 | ✅ | I + C |
| 列表查询/分页 | 可能 | 只 P(如果有性能预期) |
| 详情页展示 | 通常不激发 | - |
| 纯前端交互(tab 切换) | 不激发 | - |
| 静态展示页 | 不激发 | - |
