# Equivalence Partitioning(等价类划分)

## 核心思想

把一个输入字段的所有可能值,划分成若干"等价类"。
每个等价类内部,任何值的系统行为都应该相同。
**测试时,每个等价类只需测一个代表值就够了**——这是核心节省。

如果某个等价类的代表值通过/失败,可以推断这个类里所有值都应该有相同结果。

## 划分原则

1. **有效输入分一类(或多类)**
   - 如果有效输入有不同子类型(如 admin / regular user),每个子类型一个等价类
2. **每种无效输入分一类**
   - 不同的无效原因(空、超长、格式错、注入)各自一类,**不能合并**
   - 合并的话,失败时分不清是哪个原因导致的
3. **边界附近不算等价类**
   - 边界值由 Boundary Value Analysis 单独处理

## 套用步骤(Inspector 用)

收到测试用例后,Inspector 对每个**输入字段**做以下检查:

1. 列出该字段的所有可能等价类
2. 检查每个等价类是否至少有一个对应测试用例
3. 检查无效输入类是否被合并(常见错误)
4. 输出 P0/P1/P2 级别的反馈

## 例子:登录的 email 字段

**有效输入等价类**:
- 已注册的 email(对应 B1)
- 未注册的 email(对应 B2)

**无效输入等价类**(每类单独测):
- 空字符串
- 不含 @ 的字符串("foobar")
- 多个 @ 的字符串("a@b@c.com")
- 超长 email(>254 字符,违反 RFC 5321)
- 含中文字符的 email
- 含特殊字符可能引发注入的 email("'; DROP TABLE--@x.com")

→ 至少 8 个用例。少于 8 个就有等价类未覆盖。

## Inspector 反馈样例

```
P0-001: email 字段缺少"未注册 email"的等价类用例
- Methodology: Equivalence Partitioning
- Issue: 用例只测了已注册 email,但规约 B2 明确要求测未注册场景
- Suggested fix: 增加 TC,email 用一个不存在于数据库的地址,expected 应符合 B2

P1-001: 无效 email 等价类被合并
- Methodology: Equivalence Partitioning
- Issue: TC-005 用 "@@@" 同时违反"无 @"和"多 @"两个规则,失败时归因模糊
- Suggested fix: 拆为两个 TC,各自只违反一条
```

## 常见错误

- ❌ 把所有"无效输入"合并成一个等价类
- ❌ 漏掉某个有效输入的子类型(如 admin 用户)
- ❌ 用 Boundary 替代 Equivalence(EP 测代表值,BVA 测边界,职责不同)
- ❌ 在等价类里列了超过一个代表值(浪费,EP 的精髓是"一个就够")
