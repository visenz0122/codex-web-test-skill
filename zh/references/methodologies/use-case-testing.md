# Use Case Testing(基于用例的测试)

## 核心思想

把功能拆成用户视角的"用例"(use case),每个用例描述一个完整的业务流程。
每个用例都要测三类路径:

- **主路径(Main Flow)**:用户最常走的成功流程
- **备选路径(Alternate Flow)**:同一目标的不同实现方式
- **异常路径(Exception Flow)**:失败、超时、资源不足等

这是 Cartographer 在生成用例时**必须自检**的覆盖维度
(对应 SKILL.md 里说的"职责 b 内嵌"),Inspector 复查这个覆盖是否完整。

## 套用步骤(Inspector 用)

1. 对规约的每个 behavior 或 use case,检查三类路径的用例数
2. 三类都至少 1 个 → 通过
3. 缺少异常路径 → 几乎必然提 P0(异常路径漏测是最常见的 bug 来源)
4. 缺少备选路径 → 视情况 P1 或 P2

## 三类路径的判别

**主路径**:
- 最直接的成功流程
- 用户最频繁走的
- 通常 1 个 use case 对应 1 个主路径

**备选路径**:
- 不同的成功方式(达到同样目标但走不同路径)
- 例子:登录可以用账号密码、可以用 SSO、可以用魔法链接——这是 3 个备选

**异常路径**:
- 失败的所有原因
- 超时、网络错误、权限不足、资源耗尽
- 例子:登录失败可能因为密码错、账号锁定、服务器 500、网络断、CAPTCHA 不通过

## 例子:登录功能

**主路径**:
- 已注册用户输入正确密码 → 登录成功

**备选路径**:
- SSO 登录(如 Google / GitHub)
- 魔法链接(邮件登录)
- 二步验证后登录

**异常路径**:
- 密码错误
- 账号不存在
- 账号已锁定
- 限流触发
- 网络断开
- 后端 500
- CAPTCHA 失败

→ 至少 11 个用例。少于这个数,Inspector 必然有话说。

## Inspector 反馈样例

```
P0-001: 异常路径覆盖不足
- Methodology: Use Case Testing
- Issue: 用例只测了密码错误,没测账号锁定、限流、网络异常
- Affected: 规约 behavior B4(限流)、B5(账号锁定)有但无对应用例
- Suggested fix: 增加 TC 覆盖每个异常路径

P1-001: 备选路径未测
- Methodology: Use Case Testing
- Issue: 规约提到支持 Google 登录,但用例只测了密码登录
- Suggested fix: 增加 SSO 登录的 TC,或在 out_of_scope 中明确声明本期不测
```

## 严重等级判定

- **P0**:主路径缺失(几乎不可能,但有的话直接 P0)
- **P0**:核心异常路径(权限/资源/数据完整性相关)缺失
- **P1**:备选路径缺失(用户体验受影响,但功能本身可用)
- **P2**:异常路径中较少见的(如网络抖动)未测

## 常见错误

- ❌ 只测主路径(典型的 happy path 偏好)
- ❌ 把"输入校验失败"当成异常路径的全部(那只是输入相关的异常)
- ❌ 服务器侧异常(500、超时)忽略
- ❌ 备选路径的"成功"被合并到主路径(失去单独验证的机会)
