# 前端渲染保真度

## 适用范围

任何涉及"后端数据 → 前端展示"的功能。判别要点:**功能里有"数据被前端渲染给用户看"这个动作**。

具体场景:

- 任何展示用户输入的功能(消息、评论、文章、用户名等)
- 任何展示后端时间戳的功能(创建时间、更新时间、活动日历)
- 任何展示数字 / 货币的功能(金额、计数、百分比、统计图表)
- 任何展示 Markdown / 富文本 / HTML 内容的功能
- 任何展示多语言字符 / emoji / 特殊符号的功能
- 任何对长文本做截断 / 高亮 / 摘要的功能
- 任何展示文件 / 图片 / 链接的功能
- 任何"实时格式化"的功能("3 分钟前"、"刚刚"、"昨天"等相对时间)

**核心问题**:这些场景里,后端数据正确 ≠ 用户看到的正确——前端渲染层可能丢失、转义、格式化错误。
后端工程师测后端 API 看不到这种 bug;前端工程师测 UI 容易忽略细节差异;**只有 E2E 测试同时看两端,才能抓到**。

## 必查清单

### Markdown / 富文本渲染

- 粗体 / 斜体 / 下划线:`**bold**` / `*italic*` 是否生成 `<strong>` / `<em>` 元素
- 标题:`# H1` 到 `###### H6` 是否生成对应级别 heading
- 列表:有序 / 无序列表 / 嵌套列表是否正确缩进
- 代码块:` ``` ` 包裹的代码是否带语法高亮 / monospace 字体
- 行内代码:`` `code` `` 是否生成 `<code>` 元素
- 链接:`[text](url)` 是否生成可点击的 `<a>` 标签 + 目标正确
- 图片:`![alt](src)` 是否生成 `<img>` + alt 文本可达
- 表格:Markdown 表格是否渲染为 HTML table
- 引用块:`> quote` 是否生成 `<blockquote>`
- 分割线:`---` 是否生成 `<hr>`

### HTML 转义安全

- **脚本注入防护**:输入 `<script>alert(1)</script>` 应显示为字面字符串,**不**执行
- **属性注入防护**:输入 `<img src=x onerror=alert(1)>` 应被过滤或转义
- **链接协议白名单**:`javascript:` / `data:` 等危险协议是否被拒
- **保留 HTML 实体**:用户输入 `&amp;` 应显示为 `&`(而不是 `&amp;` 字面),前提是这是预期行为

### 时间 / 时区

- **UTC → 本地时区**:后端存 UTC,前端按用户时区显示是否正确
- **DST 切换**:夏令时切换日的时间显示是否正确
- **跨日界**:后端时间戳处于本地零点附近时,日期是否正确(不是显示前一天 / 后一天)
- **相对时间**("3 分钟前" / "刚刚" / "昨天"):计算逻辑是否正确,临界点是否准确
- **时间格式**(YYYY-MM-DD HH:mm vs YYYY/MM/DD 等):是否符合用户区域设置

### 数字 / 货币

- **千分位**:1234567 → "1,234,567" 是否生效
- **小数精度**:12.3456 显示几位?保留位数是否符合预期(财务通常 2 位)
- **数字截断 / 四舍五入**:1234.567 显示为 "1234.57" 还是 "1234.56"?是否丢失精度
- **货币符号**:"¥" / "$" / "€" 是否正确,位置(前 / 后)是否正确
- **大数缩写**:1000000 显示为 "1M" / "100 万" 等
- **百分比**:0.1234 显示为 "12.34%" 还是 "0.1234%"
- **科学计数法**:超大 / 超小数字处理

### 字符编码

- **emoji**:🎉 是否正确显示(不是 ?? 或方块)
- **emoji 组合**:👨‍👩‍👧‍👦(家庭 emoji,ZWJ 序列)是否完整,不被拆成多个字符
- **CJK 字符**:中日韩字符是否正常显示
- **特殊符号**:① ② ③、上下标、数学符号
- **零宽字符**(U+200B 等):是否被正确处理(显示为零宽 / 或被过滤)
- **从右到左文字**(阿拉伯文 / 希伯来文):方向是否正确
- **组合字符**(如 ñ = n + 组合波浪):是否正确显示

### 长文本处理

- **截断显示**:超长字符串是否被正确截断(不超过容器宽度)
- **省略号位置**:截断标记 "..." 是否在合适位置(末尾 / 中间 / 智能)
- **截断后的完整版访问**:点击 / 悬停 / 展开能否看到完整内容
- **多行折行**:长单词 / URL 是否会撑开容器
- **CSS overflow**:scroll / hidden / ellipsis 是否符合预期

### 数据格式化转换函数(易出 bug)

- **JSON.stringify / parse**:嵌套对象 / 数组 / null 处理
- **Boolean 显示**:true/false 显示为 "是/否" / "✓/✗" / "true/false" — 是否一致
- **空值处理**:null / undefined / 空字符串显示为 "—" / "(空)" / 留白 — 是否一致
- **数组 / 对象渲染**:数组 [1,2,3] 显示为 "1, 2, 3" 还是 "[1,2,3]" 还是逐项展示

### 实时计算 / 动态字段

- **计数器更新**:列表新增/删除时,顶部计数(如"共 X 条")是否同步更新
- **统计字段**:总额 / 平均值 / 比例等实时计算是否准确
- **状态徽章**:数据状态变化时,UI 徽章是否同步切换
- **进度条**:数值变化时,百分比 / 进度条 / 颜色是否同步

## 怎么测(必须配合后端 verify + Agent 截图判断)

**核心原则**:**前后端必须同时观察并对比**。单侧断言抓不到这类 bug。

### 推荐 Operator-mode:C(混合模式)

匹配本场景模式的 TC,**几乎都应该标 Operator-mode: C**——
因为渲染保真度同时需要"后端数据正确性"(Playwright 精确 SQL/API 断言)和"前端渲染视觉判断"(LLM 看截图语义理解)。
单纯 Playwright 用 DOM 选择器检查 `<strong>` / `<h1>` 元素能抓部分 bug,但**抓不到这些**:

- emoji 是不是显示成方块 / 问号(DOM 里是 emoji 字符,但实际渲染失败,字体不支持)
- 时区文本"2026-04-26 18:00"看起来对,但**和当前用户时区不符**(DOM 没法判断"对不对")
- 数字千分位是否符合用户区域设置(`12,345.67` vs `12.345,67`,DOM 都是字符串)
- Markdown 渲染了但样式丢失(`<strong>` 元素存在但字体没加粗,因为 CSS 失效)
- 长文本溢出气泡边界(DOM 里完整,但视觉上文字撞穿了 UI 容器)
- RTL 文字方向、CJK 字间距、emoji 组合字符(👨‍👩‍👧‍👦)拆字等

**这些都需要 Agent 看截图判断**——LLM 视觉理解擅长这种"看起来对不对"的语义问题。

### 测试用例的 Expected + Screenshot points 格式

```yaml
TC-XXX: 测发送 Markdown 消息渲染

Operator-mode: C  # 混合模式

Steps:
  1. 浏览器在 textarea 输入 "**重要**\n# 标题"
  2. 点击发送按钮
  3. 等待 SSE 流完成

Screenshot points:
  - after_step: 3  # SSE 流完成后
    save_to: screenshots/TC-XXX-after-send.png
    llm_judges:
      - "气泡内 **重要** 是否渲染为加粗(字体明显比正文粗)?"
      - "气泡内 # 标题 是否渲染为大字标题(字号明显大于正文)?"
      - "整体气泡布局是否正常,文字是否完整可见(无溢出、无被截断)?"

Expected:
  - 后端事实(SQL):messages.content = "**重要**\n# 标题"(原始 Markdown 字符串)
  - 前端渲染(浏览器):.bubble[data-id=X] 含 <strong> 和 <h1> 元素
  - 视觉断言:见 Screenshot points 的 llm_judges
```

### Agent 截图判断的工作机制

参考 SKILL.md "Operator 混合执行模式"和 `references/operator.md` §2.C:

1. **Playwright 阶段**:Cartographer 生成 Playwright 脚本,Operator 跑业务流程,
   在 `Screenshot points` 指定的步骤后调用 `await page.screenshot({path: '...'})` 保存截图
2. **LLM 后处理**:Playwright 跑完后,Operator(LLM)读保存的截图,
   对每个 `llm_judges` 问题输出判断(✅ / ❌ + 简短描述)
3. **合并报告**:Playwright trace + LLM 截图判断都写进 execution-report,
   **任一失败 = TC 综合 FAILED**

### llm_judges 的写作要求

每个判断问题必须**具体可独立回答**——LLM 看截图后能直接给 ✅ / ❌:

| ❌ 抽象问题(LLM 没法准确回答) | ✅ 具体问题(LLM 能直接判断) |
|---|---|
| "渲染是否正确?" | "气泡内 **重要** 是否渲染为加粗字体?" |
| "UI 是否合理?" | "整体气泡布局是否正常,无文字溢出?" |
| "看起来对不对?" | "时间戳显示是否符合 'YYYY-MM-DD HH:mm' 格式?" |
| "Markdown 渲染了吗?" | "# 标题 是否生成了大字标题元素(字号明显大于正文)?" |

判断问题写得越具体,LLM 判断越准——**不要写抽象的元问题**。

### 执行报告的记录方式

执行报告(`templates/execution-report-template.md`)有专门的两个段:

- **Playwright trace 摘要**:记录 Playwright 部分的 PASSED / FAILED assertion + trace.zip 路径
- **LLM 截图判断**:逐个截图 + 逐个 llm_judges 问题的 ✅/❌ + 描述
- **前后端数据对比**(可选):后端值 vs 前端渲染值 + 是否一致(进一步辅助归因)

## 关键提示

**这类 bug 在生产中特别难抓**——后端测试看不到,前端单元测试很容易漏掉边界情况(emoji、时区切换、长文本)。
**E2E 测试是最该抓这种 bug 的层**,所以这个场景模式必须主动应用。

**Agent 截图判断是不可替代的**——本场景模式的必查清单大部分项**只能通过 LLM 看截图判断**。
DOM 选择器查 `<strong>` / `<h1>` 元素能抓"渲染了吗"层面的 bug,但抓不到"渲染对不对"——
- 字体加粗了吗(CSS 失效但 DOM 元素存在,Playwright 通过)
- emoji 显示成方块了吗(DOM 是 emoji 字符,字体不支持)
- 时间显示符合用户时区吗(DOM 是字符串,Playwright 没法判断"对不对")
- 长文本溢出气泡了吗(DOM 完整,但视觉撞穿容器)

**所以匹配本场景模式的 TC 几乎必然是 Operator-mode: C**——Playwright 跑业务流程拿后端断言,
Agent 看截图判断视觉。详见上面"怎么测"段。

**不要假设"前端框架会处理这些"**——React / Vue 不自动做 Markdown 渲染、不自动做时区转换、
不自动做千分位格式化,这些都是应用层逻辑,**都可能出 bug**。

**测试数据要包含边界**:
- 字符串测试用 emoji、零宽字符、超长文本、HTML 标签字符
- 时间戳测试用零点附近、DST 切换日、跨年
- 数字测试用 0、负数、超大、小数、Infinity
- 不只测"happy path",必须测"用户实际会输入的怪东西"

## 常见叠加

通常和这些模式一起出现:
- **对话型 UI**(消息渲染、Markdown、时间戳显示几乎一定要测)
- **CRUD 列表与详情**(列表项展示、详情字段渲染)
- **个人主页 / 资料管理**(用户名、头像 alt、注册时间显示)
- **LLM agent 决策**(LLM 输出的 Markdown / 代码块渲染)
- **状态流转**(状态徽章、状态文本切换)

几乎所有"展示数据给用户看"的功能都该匹配此模式——
和"异常路径(通用)"一样,这是个**几乎总要加**的模式。

## 不适用情况

- 纯输入功能,无数据回显(如纯写入 API,后端确认即可)
- 纯静态展示页(无动态数据,内容写死在代码里)
- 纯 client_state 操作(如表单中途切换 tab,无服务器数据展示)
