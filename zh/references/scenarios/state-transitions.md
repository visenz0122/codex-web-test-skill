# 状态流转

## 适用范围

任何"系统在不同状态之间切换"的功能:

- 订单生命周期(待付 / 已付 / 发货 / 签收 / 完成)
- Token 生命周期(已生成 / 已发送 / 已使用 / 已过期 / 已撤销)
- 审批流(草稿 / 待审 / 已批准 / 已驳回 / 已撤回)
- 资源状态(空闲 / 占用中 / 释放中 / 故障)
- 任务状态(待办 / 进行中 / 已完成 / 已取消)
- 用户账户状态(正常 / 锁定 / 注销 / 暂停)
- 部署 / 发布流程

判别要点:**功能涉及一个明确的状态机**——状态有限、转移有规则、
某个状态决定了能做什么不能做什么。

## 必查清单

详细测试方法见 `methodologies/state-transition.md`。这里列要点:

### 状态可达性

每个状态都要有 TC 能进入:

- 从初始状态出发,通过合法转移路径到达每个状态
- 可达路径不止一条的状态(如 cancelled 可以从 pending 也可以从 paid 进入),
  **每条路径都要测**

### 合法转移测试

每条规约里列出的合法转移都要有 TC 测试。

### 非法转移拒绝(关键,常被遗漏)

**这是最容易漏的一类测试**:

- 已支付订单**不能**回到"待支付"
- 已签收订单**不能**回到"已发货"
- 已取消订单**不能**再支付
- 已使用 token**不能**再次使用
- 已驳回审批**不能**直接进入"已批准"

每条非法转移都应该被服务器拒绝,返回明确错误(409 / 403 / 业务码)。

### 孤岛状态和死状态

- **孤岛状态**:从初始状态出发,通过任何合法转移都到不了 → 规约设计问题
- **死状态**(陷阱):到达后没有任何转移能离开 → 终态是预期的,
  但中间状态不应该是死状态

### 状态转移触发条件

- **正常触发**:用户操作 / 系统事件 / 定时任务
- **触发条件未满足**:如"试图发货,但订单未付款" → 应被拒
- **并发触发**:两个用户同时尝试改同一状态 → 只一个成功

### 状态查询的一致性

- 列表查询时显示的状态 vs 详情查询时的状态 → 必须一致
- 状态字段在数据库 / 缓存 / UI 三处的同步

## 关键提示

**非法转移测试是 P0 漏测重灾区**——开发者直觉上"用户不会这么操作",
但攻击者会通过 API 直调或竞态条件触发非法转移。

**测"已支付订单不能再支付"这类**——很多支付系统都有此类 bug,
攻击者重放支付请求导致重复扣款 / 重复发货。

**状态机设计本身的检查**(由 Inspector 提建议给人类):
- 孤岛状态(进得去出不来)
- 死状态(到达后无法继续)
- 状态命名不清(active / pending / processing 三个差不多)

## 常见叠加

通常和这些模式一起出现:
- CRUD 列表与详情(订单等有状态资源)
- 多租户 / 权限矩阵(状态变更的权限)
- 异步 / 流式输出(状态变更可能触发流式通知)

## 不适用情况

- 无状态的纯查询功能(没有状态机)
- 单一状态的功能(永远只有一个状态)
- 状态转移规则不固定的功能(如自由编辑文档)
