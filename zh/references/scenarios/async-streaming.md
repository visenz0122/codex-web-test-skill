# 异步 / 流式输出

## 适用范围

任何"发起请求 → 等待 → 渐进出现"的功能:

- SSE(Server-Sent Events)流式响应
- WebSocket 推送
- 长轮询
- 流式 LLM 输出(逐字 / 逐 token 渲染)
- 文件上传进度 / 下载进度
- 后台任务进度显示
- 实时通知 / 推送

判别要点:**响应不是单次完整返回,而是分多次或逐步到达**。

## 必查清单

### 超时与跳出(硬要求)

**每个等待步骤必须有 timeout 和跳出条件——绝不允许"无限等"**。

- 流式输出的最大等待时间(如 60s)
- 等待中无新内容到达的"心跳超时"(如连续 10s 无新数据)
- 超时后的处理:报错、提示重试、关闭连接

### 正常流程

- 发起 → 接收(分多次)→ 完整结束 → 后续状态正确
- 流的"开始信号"(如 SSE 的 connection event)
- 流的"结束信号"(如 SSE 的 [DONE] 或 close event)
- 流的内容拼接是否正确(按 token / chunk 拼成完整结果)

### 提前取消

- **用户主动取消**:点取消按钮、关闭对话框、跳走页面
- **取消后后端是否立刻停止**:不应继续浪费计算资源
- **取消后已收到内容如何处理**:保留还是清空(取决于业务)

### 网络异常处理

- **网络断开**:是否报错 / 自动重连 / 缓冲已收到内容
- **网络抖动**:连接断了又恢复,是否能续接
- **慢网络**:每段延迟很大,UI 是否正常显示进度
- **代理 / VPN 中断流式连接**:很多企业网络会缓冲完整响应才发,流式实际不流式

### tab 行为

- **切走 tab 是否暂停接收**:浏览器节流(throttle)行为
- **切回 tab 是否能恢复**:能否显示已经发生但没看到的内容
- **后台 tab 持续接收的资源消耗**

### 并发请求

- **同时发起两个流式请求**:是否互不干扰
- **上一个还没完就发起新的**:旧的是否被正确取消、状态是否清晰
- **请求竞态**:慢请求和快请求并发,显示的是哪个的结果

### 错误恢复

- **流中途出错**(后端 500):优雅降级,显示已收到部分 + 错误提示
- **部分内容已收到时出错**:已收到的部分是否保留 / 显示
- **重试机制**(若有):是否从断点续传 / 还是从头重来

### 资源清理

- **取消后,后端连接 / 计算资源是否释放**
- **长时间不响应**:是否触发心跳保活、还是就让它挂着
- **客户端内存**:长流式输出是否导致前端内存泄露

## 关键提示

**异步场景的 expected 必须用 "eventually" 表达最终稳定状态**,不能写"立即"。
所有等待步骤必须显式给出 timeout——这是设计层面的硬要求,不能漏。

**死循环陷阱**:测试用例如果写"等草案 → 点确认 → 等推理回贴",但没写 timeout 和跳出条件,
LLM 这一轮没产出预期内容时,Operator 会一直等 → 死循环。

**模拟网络异常**在某些测试工具下很难做到——Browser Use 通常不能精确模拟"网络断开"。
如果用例需要测这点,通常需要 Playwright + offline mode,或者标 manual_upload。

## 常见叠加

通常和这些模式一起出现:
- 对话型 UI(几乎一定)
- LLM agent 决策(LLM 输出几乎都是流式)
- 文件上传 / 下载(上传 / 下载进度也是异步)
