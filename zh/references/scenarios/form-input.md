# 表单输入型

## 适用范围

任何"填表 + 提交"模式:

- 登录、注册、密码重置
- 提交资料、申请单、问卷、反馈表单
- 创建资源(创建账号、创建订单、发布文章)
- 设置 / 配置类功能(修改通知偏好、修改密码)

判别要点:**有一个或多个输入字段,用户填写后通过"提交"动作触发系统行为**。
单次操作,不是多步流程。

## 必查清单

### 输入字段层面

- **每个字段的等价类**:有效输入(可能多类)+ 各类无效输入(空、超长、格式错、特殊字符)
- **每个字段的边界值**:长度上下限、数值上下限、字符集边界
- **跨字段约束**:字段 A 和 B 的组合校验(如 country=US 时 zipCode 必填)

### 客户端 vs 服务器

- **客户端校验是否独立有效**:删除前端校验代码后,服务器还能拒绝吗?
- **绕过客户端校验直接提交**:用 API 调用、改 DOM 后提交、用 curl 直接发请求
- **错误提示是否一致**:客户端报错文案和服务器报错文案是否对应

### 安全相关

- **CSRF / token 校验**:伪造 token、用过期 token、不带 token 提交
- **特殊字符注入**:SQL 注入特征字符、XSS payload、Markdown 注入、零宽字符、URL 注入
- **HTML 标签输入**:`<script>` / `<img onerror=...>` 等是否被转义

### 提交行为

- **重复提交防护**:连点提交按钮、API 重放、断网重连后自动重试
- **提交中状态**:按钮置灰、loading 显示、避免双发

### 用户体验

- **保存草稿 / 中途离开**(有这个功能时):未保存提示、自动保存恢复
- **提交成功后**:跳转目标、清空表单、显示成功提示

## 关键提示

**绝对不要把"客户端校验"当成全部校验**。客户端只能改善用户体验,不能作为安全屏障——
任何安全相关的输入校验必须在服务器端独立测试。

**不要把多个无效原因合并成一个 TC**。比如 "@@@" 同时违反"无 @"和"多 @"两个规则,
失败时归因模糊,要拆成两个 TC 各自只违反一条。

## 不适用情况

- 多步流程类(如订单结算的多步)→ 用"用户旅程 / Use Case Testing" 方法论
- 对话型输入(虽然也有输入框)→ 用"对话型 UI"模式
- 文件上传为主 → 用"文件上传 / 下载"模式
