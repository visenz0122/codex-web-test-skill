# 对话型 UI

## 适用范围

"输入框 + 发送 + 历史消息" 模式:

- 聊天 / 客服窗口
- AI 助手 / chatbot
- 评论 + 回复
- 即时消息(IM)
- 论坛 / 帖子讨论
- AI agent 对话界面

判别要点:**用户通过输入框反复发送消息,系统的输出和历史消息一起展示**。
特征是"多轮、有历史、强交互"。

## 必查清单

### 输入边界(输入框接受的文本)

测试输入字段允许什么样的内容:

- **正常文本**:中文、英文、中英混排
- **emoji / 特殊符号**:🎉、各种 emoji 组合
- **标点符号**:全角标点、半角标点、混用
- **零宽字符 / 不可见字符**:U+200B、U+FEFF 等
- **超长文本**:贴一整篇文章(检查截断或拒绝行为)
- **空白**:纯空格、纯换行、Tab 字符
- **极短**:1 个字符、单个标点

### IME 输入法状态(中文 / 日文 / 韩文)

这是**最容易遗漏**的一类:

- **IME 半成品状态**:候选词浮窗中按 Enter,是否会误发送拼音 / 假名半成品
- **IME 候选选择中按 Backspace**:行为是否符合预期
- **从其他应用粘贴含 IME 状态的内容**

### 发送行为

- **Enter 发送 vs Shift+Enter 换行**:行为是否符合预期且与 placeholder 文案一致
- **双击发送按钮**:防双发,只发送一次
- **连按 Enter 两次**:防双发,只发送一次
- **网络断开时点发送**:进入待发送队列?直接报错?

### 多轮上下文

- **第二轮指代第一轮**:用"它"、"刚才那个" 等代词,系统能正确解析吗(尤其对 LLM agent)
- **历史消息超长时的截断策略**:是否丢失关键信息
- **超长会话的滚动 / 性能**:历史消息很多时 UI 是否还流畅

### 中途修改

- **输入到一半粘贴大段文本**:是否正确接受
- **输入到一半切到其他 tab 再回来**:输入框内容是否保留
- **输入到一半浏览器刷新**:草稿是否恢复(若有此功能)
- **修改未发送的内容**:删除部分 / 重写

### 会话恢复

- **tab 切走再回来**:流式输出是否恢复 / 续接
- **浏览器关闭重开**:历史是否还在(取决于服务器存储)
- **session 中途过期**:用户输入到一半 session 过期,如何处理

### 撤销 / 编辑已发送消息(若有)

- **撤销时序窗口**:发送后多久内可撤销
- **编辑后版本展示**:编辑前 vs 编辑后,UI 如何区分
- **被编辑消息的引用**:已被回复的消息编辑后,引用是否还指向原内容

### 注入相关(对话内容会被存储或反显时)

- **HTML 标签**:输入 `<script>` 是否被转义
- **Markdown XSS payload**:`[click](javascript:alert(1))` 等
- **协议字符串伪造**:输入 `__SYSTEM__:` / `__CONFIRMED__:` 等业务协议字符串
- **URL 注入**:输入恶意 URL

## 关键提示

**对话型 UI 的"等价类"和登录表单完全不同**。
- 登录的等价类围绕"账号是否存在 / 密码是否正确"
- 对话框的等价类围绕"输入字符种类 / 输入时机 / 输入长度"

不要照搬登录的 EP 思路到对话框——会写出大量无意义的"中文 / 英文 / 数字"等价类用例。

**IME 状态在自动化测试工具下很难精确触发**——大多数浏览器自动化工具(Playwright、Claude in Chrome)
都不能模拟 IME 半成品状态。如果用例必须测这点,通常需要标 manual_upload(让人类手动测一次)。

**多轮上下文是 LLM agent 的核心能力测试**——不要只测单次问答。

## 常见叠加

通常和这些模式一起出现:
- 异步 / 流式输出(对话型 UI 几乎一定有流式)
- LLM agent 决策(若是 AI 对话)
- 用户认证 / 会话管理(对话历史绑定用户)
- 多租户 / 权限矩阵(只能看自己的对话历史)
