# 多租户 / 权限矩阵

## 适用范围

任何涉及"用户、角色、资源"隔离的系统:

- 多角色系统(admin / user / guest 等)
- 多租户 SaaS(组织 A 的数据租户 B 不能看)
- 权限分组(部门、团队、项目权限)
- 资源所有权(用户只能管理自己创建的资源)
- 共享资源 / 公开资源(明确不需要登录就能访问的部分)

判别要点:**功能涉及"谁能做什么"的判断**,而不只是"功能本身怎么做"。

## 必查清单

### 角色权限矩阵

**穷举**所有角色 × 所有操作:

```
角色 \ 操作    | Read | Write | Delete | Share | Admin
admin          | ✅   | ✅    | ✅     | ✅    | ✅
user           | ✅   | 自己  | 自己   | 自己  | ❌
guest          | ✅   | ❌    | ❌     | ❌    | ❌
```

**每个 ✅ 测正路径**(成功)。**每个 ❌ 测被拒路径**(返回 403 / 跳转登录)。

不要只测自己角色的正路径——典型的开发者偏向。

### 跨租户隔离

- **租户 A 的用户能否看到租户 B 的资源**:列表查询、详情访问、API 直调
- **租户 A 的用户能否修改租户 B 的资源**:URL 改 ID、API 直调
- **跨租户搜索**:租户 A 搜索时,租户 B 的内容会泄露吗

### 越权访问途径

不只是 UI 层面,所有可能的越权途径:

- **URL 改 ID**:`/profile/123` 改成 `/profile/124` 直接访问
- **API 直调**:绕过 UI 权限检查,直接调用 API
- **改 cookie / localStorage 中的 role 字段**:把 role=user 改为 role=admin
- **未保护的 admin 接口**:有的开发者忘记给某些 admin API 加权限校验
- **改请求 body 里的 user_id**:服务器是否信任客户端传的 ID 而不是 session 里的

### 权限提升攻击

- **普通用户改 role 字段**:登录后通过 API 修改自己的 role,看是否被拒
- **借助管理员功能**:能否利用某个 admin 暴露的接口提升自己权限
- **Race condition**:在权限被撤销的同时执行高权限操作

### 权限失效

- **用户被降级 / 移除角色后**:旧 token / 旧 session 是否立刻失效
- **管理员撤销用户权限**:被撤销用户的进行中操作如何处理
- **角色定义变更**:role X 不再有 permission Y 后,持有 X 的用户立刻失去 Y 吗

### 共享资源 / 公开资源

- **未登录访问公开资源**:能正常看吗
- **分享给某人**:只有该人能看,其他人不能看
- **取消分享**:被取消的人立刻失去访问权
- **分享链接**:Token 链接的过期、撤销、单次使用等

## 关键提示

**权限矩阵是 P0 漏测的"重灾区"**——这是最严重的安全漏洞类型。
任何一格 ❌ 测试缺失,都可能在生产中变成 CVE 级别的越权 bug。

**测"被拒"比测"成功"更重要**。开发者直觉上写 happy path,
但安全 bug 几乎都在"应该被拒但没被拒"上。

**不要假设服务器一定会校验**——尤其是后端开发者经常假设"前端会过滤",
然后写出了"信任客户端 user_id"的代码。所有越权测试都必须**绕过 UI** 直接打 API。

## 常见叠加

通常和这些模式一起出现:
- CRUD 列表与详情(权限通常作用在 CRUD 上)
- 用户认证 / 会话管理(权限基于身份)
- 个人主页 / 资料管理(谁能看谁的资料)
