## Resource Dependency Matrix

<!--
Cartographer phase 2 must fill — list shared resources + which TCs break + which depend,
used to detect circular dependencies / deadlock between TCs.

What is "shared resource": used by multiple TCs. Examples:
- Test users (user_test, admin_test)
- Test data (orders, articles, tokens)
- Global state (quotas, counters)
- External state (sent emails, webhook calls)

Why list: typically only one user_test, few test orders locally.
If TC-A deletes user_test but doesn't teardown rebuild, TC-B needing user_test fails.
This matrix makes circular dependencies visible at a glance — Inspector sees one table to detect issues.

Judgment method:
- Destructive TC: Destructive: yes TC modifies / deletes / consumes resources
- Dependent TC: Preconditions / Setup actions need resource in certain state
- Does it have teardown recovery: does destructive TC's Teardown actions actually recover resource?
-->

| Shared resource | Destructive TC | Dependent TC | Teardown recovery? | Notes |
|---|---|---|---|---|
| user_test | TC-005 (delete) | TC-001, TC-002, TC-008 | ✓ TC-005 teardown rebuilds | Closed loop |
| order_001 (state: pending) | TC-D (cancel → cancelled) | TC-E (list pending orders), TC-F (pay) | ⚠ terminal state, cannot teardown | Use separate order_002 |
| email_service | TC-password reset (send email) | none | ✗ irreversible | Mock email service, see spec §3.5 |

**Matrix interpretation**:

- All ✓ → test suite closed loop, any order run without pollution
- Has ⚠ → resource has circular dependency risk, needs solution (separate resource / mock / rebuild)
- Has ✗ → irreversible operation, **must** use mock or declare in spec §3.4b engineering boundary
